## 技术记忆候选

> 只记录未来可能复用的工程事实。无则写“无”。

- 架构 / 接口决策：`launch.sh` 的启动链固定为三分支——先复用通过 `http://127.0.0.1:{port}/json/version` 终检的现有 CDP；无 CDP 且主进程运行时进入交互确认；主进程未运行时正常调试启动。应用、Node、端口和进程生命周期能力继续集中在 `lib/runtime.sh`，`inject-current.sh` 复用同一 CDP 就绪判断。
- 工程规则 / 工作流：显式 `CODEX_APP` 与自动发现候选都必须通过 bundle id `com.openai.codex` 终检；启动端口必须在任何进程探测、确认、终止或 `open` 前验证为十进制 `1..65535`；真实强制重启属于破坏性人工验收，不应在承载任务的 ChatGPT/Codex 会话中自动执行。
- 实现模式 / 复用写法：Info.plist 身份标量先写入权限固定为 `0600` 的精确临时文件，在进入 Bash 变量前用 `od` 检查原始 NUL，再验证单行控制字符；`CFBundleExecutable` 还需拒绝选项形名称和按 C locale 计算超过 19 字节的值，之后转义为字面 ERE，才可用于 `pgrep -x` / `pkill -KILL -x`。
- 易错点 / 修复策略：Shell 命令替换会剥离尾随换行并丢弃 NUL，不能直接读取安全敏感的 plist 标量；macOS 非 `-f` 的 `pgrep` / `pkill` 存在 19 字节短进程名边界；`pgrep -x` 的 pattern 仍是正则且状态 2/3 不是“未运行”；`read` 必须保留首尾空白，只有原始值恰为 `Y` 或 `y` 才能进入强杀分支；注入失败必须显式传播，不能依赖条件上下文中的隐式 `errexit`。
- 验证经验：破坏性启动链可通过替换 TTY、输入和底层系统命令进行零真实副作用的函数级与 launch 级回归；成功路径必须显式断言状态 0，失败用例必须同时断言没有调用 `pgrep`、`pkill`、`open` 或注入器，避免条件上下文抑制 `errexit` 形成假绿。
