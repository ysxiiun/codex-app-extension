## Technical Memory Candidates

> Only record engineering facts with future reuse value. Write "none" if none.

- Architecture / Interface Decisions: 页面浮层按职责分为三类：`menu/listbox` 瞬态交互浮层不参与 right rail；composer 上方 `bottom-full` 附着组件保留内部 native width reset，但用独立的 `--codex-app-extension-aligned-overlay-offset-x` 跟随 composer；持久 `thread-floating-content` rail 继续参与主内容避让并保持局部零偏移。
- Engineering Rules / Workflows: 浮层分类变化必须同步 `buildDiagnoseSource()`、`README.md` 与 `.easy-coding/ABSTRACT.md`；诊断字段使用 `transientInteractiveOverlayCandidates` 和 `composerAttachedOverlayCandidates` 暴露命中目标、对齐属性、偏移和坐标。
- Implementation Patterns / Reusable Approaches: 在 `findRightFloatingRail()` 候选入列前排除瞬态菜单与 composer 附着组件；附着 wrapper 以 `bottom-full` 加 composer top menu 关联信号识别，使用专用属性幂等标记，离开 Codex surface 或关闭增强时清理；已有原生 transform 时跳过横向位移，避免覆盖动画。
- Pitfalls / Fix Strategies: 通用 `absolute/fixed/sticky` 几何启发式会把短暂菜单当作 rail，形成“宽度回写—菜单重定位—再测量”的反馈环；把所有 `bottom-full` 统一清零 content offset 又会让任务/Git 顶部组件停留在增强前中心。修复不能靠延长 debounce、冻结宽度或整体移动所有浮层。
- Verification Experience: `./verify.sh` 可用正负例覆盖自身/祖先 menu、后代 listbox、composer 附着目标、持久 rail、独立变量不被 native reset 清零及诊断源码；真实视觉仍应在线确认菜单稳定、顶部组件同中心和右侧 rail 避让。
