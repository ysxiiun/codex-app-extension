## Technical Memory Candidates

> Only record engineering facts with future reuse value. Write "none" if none.

- Architecture / Interface Decisions: 瞬态交互浮层分类围绕同一个 `TRANSIENT_INTERACTIVE_OVERLAY_SELECTOR` 建立对称闭包：元素自身 `matches()`、位于菜单内部的 `closest()`、包裹菜单的 `querySelector()` 均使用完整的 `role=menu/listbox` selector；任一命中即在几何测量前从 right rail 候选排除。
- Engineering Rules / Workflows: 分类实现、`buildDiagnoseSource()` 与 README 必须保持同一 menu/listbox 语义；诊断字段 `transientInteractiveOverlayCandidates` 同时展示可见 role 节点和最近的可见 `fixed/absolute/sticky` 定位 wrapper，并按元素去重。
- Implementation Patterns / Reusable Approaches: 对 DOM 角色节点与真正参与布局几何扫描的定位祖先同时取证；诊断作用域内无法复用 installer 局部函数时，只内联最小的 layout-shell 停止选择器，避免把候选扩散到更高层布局容器。
- Pitfalls / Fix Strategies: 只覆盖“自身 menu/listbox、位于 menu 内、后代 listbox”会遗漏真实 `fixed wrapper > role=menu` 结构；该 wrapper 若同时满足右侧锚定、窄宽和高度条件，会在下一轮布局刷新中被选为 rail，使 composer 从 `564px` 缩到 `303px`。修复应补齐 selector 方向，而不是冻结宽度、延长 debounce 或关闭菜单动画。
- Verification Experience: 从生成 installer 源提取分类函数，用最小 HTMLElement stub 覆盖 menu/listbox 自身、祖先、后代正例，以及普通持久 rail 和非 HTMLElement 负例；必须保留“后代 role=menu”这一旧实现必失败的真实回归。对称后代匹配仍有“持久面板内部长期挂载可见菜单”被误排除的残余风险，应通过逐元素判定、layout-shell 边界和在线诊断约束。
